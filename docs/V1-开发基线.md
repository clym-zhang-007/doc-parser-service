## 项目目标
搭建可运行的多格式文档解析 API 服务（工业化第一版），先实现核心链路：

上传文档 → 创建异步任务 → 解析执行 → 查询状态 → 获取标准化结果。

## 当前技术栈（V1）
- API：FastAPI
- 异步任务：Celery + Redis
- 数据库：PostgreSQL
- 解析核心：LlamaIndex（先主用）
- 文件存储：本地 `storage/`（后续可切 MinIO/S3）

## V1 功能范围（In Scope）
1. 支持格式：`PDF / DOCX / TXT / Markdown`
2. 核心接口：
   - `POST /v1/jobs`
   - `GET /v1/jobs/{job_id}`
   - `GET /v1/jobs/{job_id}/result`
   - `GET /health`
3. 统一结果 JSON（document/blocks/meta/error）
4. 异步状态机：`queued/running/success/failed`
5. 基础日志与错误码

## V1 非目标（Out of Scope）
- 多租户计费
- 复杂 OCR 优化
- 向量库与检索问答
- 管理后台
- LangChain 复杂编排（仅预留扩展点）

## 结果 JSON 与分块策略（可追溯）

成功任务写入 `jobs.result_json`，其中 `meta` 用于追溯「全文如何抽取」与「如何切成 blocks」：

| 字段 | 含义 |
|------|------|
| `parser` | 全文抽取路径（如 `llamaindex.SimpleDirectoryReader` 或回退 `utf8_plain` / `pypdf.PdfReader` / `python-docx`） |
| `blocks_schema_version` | 分块 schema 版本，与 `app/services/document_parse.py` 中 `BLOCKS_SCHEMA_VERSION` 一致（当前 `1.6`） |
| `blocks[].char_count` | （`1.4`）块内主体字符数，与 `text` 字符串长度一致，便于预估 token |
| `structure_quality` | （`docx.structure_v1_1`）结构识别质量摘要：`style_hit_ratio` / `numbering_hit_ratio` / `heuristic_hit_ratio` / `table_hit_ratio` / `avg_confidence` |
| `block_strategy` | 分块策略标识，与 `parser` 独立；策略变更时改此字符串或升 `blocks_schema_version` |

**当前 `block_strategy` 取值（与代码一致）**：

- `markdown.heading_list_v4`：在 v3 基础上，GFM **管道表**为 `table`（`rows` 为单元格矩阵）；**整行**仅 `![alt](url)` / `[label](url)` 为 `image` / `link`；行内混排仍属 `paragraph`。其余：ATX 标题、段落、围栏代码、`separator`、`blockquote`、列表合并规则同 v3，再经小块合并与过大块切分。
- `txt.paragraph_v1`：`txt` 软换行修复后按段落分块，再经合并/切分。
- `docx.structure_v1_1`：`docx` 使用 `python-docx` 结构化解析为 `heading` / `paragraph` / `list_item`→`list` / `table`，并给结构块附 `source` + `confidence`；样式缺失时用启发式兜底。
- `simple.blankline_v1`：`pdf` 等暂未定制时，按空行分段并做合并/切分兜底。

实现位置：`app/services/document_parse.py`（`split_blocks()`）。
