# 文档解析 API 服务（V1）

## 项目目标
搭建可运行的多格式文档解析 API 服务（工业化第一版），先实现核心链路：

上传文档 → 创建异步任务 → 解析执行 → 查询状态 → 获取标准化结果。

## 当前技术栈（V1）
- API：FastAPI
- 异步任务：Celery + Redis
- 数据库：PostgreSQL
- 解析核心：LlamaIndex（先主用）
- 文件存储：本地 `storage/`（后续可切 MinIO/S3）

## V1 功能范围（In Scope）
1. 支持格式：`PDF / DOCX / TXT / Markdown`
2. 核心接口：
   - `POST /v1/jobs`
   - `GET /v1/jobs/{job_id}`
   - `GET /v1/jobs/{job_id}/result`
   - `GET /health`
3. 统一结果 JSON（document/blocks/meta/error）
4. 异步状态机：`queued/running/success/failed`
5. 基础日志与错误码

## V1 非目标（Out of Scope）
- 多租户计费
- 复杂 OCR 优化
- 向量库与检索问答
- 管理后台
- LangChain 复杂编排（仅预留扩展点）

## 统一错误码（V1）
- `UNSUPPORTED_FILE_TYPE`
- `FILE_TOO_LARGE`
- `JOB_NOT_FOUND`
- `RESULT_NOT_READY`
- `PARSE_FAILED`
- `INTERNAL_ERROR`

## 开发里程碑（建议 10 天）
- Day 1-2：骨架 + compose + 配置
- Day 3-4：任务状态机与 API
- Day 5-6：解析器接入（LlamaIndex）
- Day 7：重试与错误处理
- Day 8：测试与样例回归
- Day 9：日志与基础指标
- Day 10：验收与打标发布

## 验收标准（DoD）
1. `docker compose up` 后 API 可用
2. 4 种格式可完成异步解析
3. 状态流转完整且可查询
4. 失败任务有可追踪错误码
5. 最小测试集通过

## 变更规则（防止目标漂移）
- 任何新增能力先写入 `docs/DECISIONS.md`，标记：背景、收益、成本、是否进入 V1。
- 若新增内容不在 In Scope，默认延期到 V1.5/V2。
- API 契约优先稳定，不随内部重构变更。
